# Runbook — Bitrix24 MCP Bridge

Как установить, настроить мост и пользоваться им дальше.

Мост даёт локальному ИИ-агенту доступ к твоим задачам, проектам и чатам Bitrix24 (чтение
и — для вызовов из каталога — выполнение действий) — в объёме прав твоего пользователя,
через **уже открытую браузерную сессию**. Без прав администратора и без официального
REST-вебхука.

---

## Как это устроено (чтобы был понятен порядок запуска)

```
ИИ-агент ─stdio─► MCP-клиент ─UDS(bridge.sock, 0600)─► daemon ─WS(127.0.0.1:39917, токен+Origin)─► Расширение (вкладка портала)
                                                                                                     │  fetch + свежий sessid + cookie
                                                                                                     ▼
                                                                                         Bitrix24 (задачи / группы / чаты)
```

Для живого запроса нужны **одновременно** три вещи:
1. запущенный **daemon** (его поднимает сам MCP-клиент при первом обращении);
2. открытая **залогиненная** вкладка портала;
3. установленное расширение на этой вкладке.

Ключевые принципы:
- **Один daemon держит порт `39917`, все клиенты общаются с ним через UDS.** N клиентов и
  health-проб сосуществуют — конфликта порта больше нет. Daemon завершается по простою (~5 мин).
- **`actions.json` — это данные и allowlist одновременно.** Что не описано — вызвать нельзя.
  Каталог может включать мутирующие вызовы (`…add`/`…update`/`…delete`/…) — они загружаются
  и выполняются наравне с read-only; allowlist — единственная граница.
- **daemon слушает только `127.0.0.1`**, требует общий токен и Origin вкладки из набора порталов.
- **Свежесть = момент запроса.** Подписки на новые сообщения нет.

---

## Требования

- **Node.js** с `npm` (для установки CLI). Bun — только для разработки, конечному пользователю не нужен.
- **Chromium/Chrome** (мост работает через loopback-исключение для `ws://127.0.0.1` из HTTPS).
- Аккаунт Bitrix24, под которым ты залогинен в браузере.

---

## Установка и настройка (разово)

### 1. Настроить портал

Без глобальной установки — через `npx` (первый запуск скачает пакет и закэширует):

```bash
npx -y bitrix24-mcp-bridge setup
```

`npx` резолвит бинарь напрямую (пакет `bitrix24-mcp-bridge` объявляет `bin` под этим
именем), собирать `dist/cli.js` вручную не нужно — он уже собран и опубликован в пакете.

Альтернатива — глобальная установка (быстрее стартует MCP-клиент, т.к. не гоняет
резолв `npx` при каждом запуске):

```bash
npm i -g bitrix24-mcp-bridge
bitrix24-bridge setup
```

**Первый запуск** (нет конфигурации) спросит:
- **origin портала** — например `https://acme.bitrix24.ru` (без пути и слэша);
- **alias** — короткое имя (по умолчанию — из хоста).

После этого он автоматически:
- сгенерирует общий токен (`crypto.randomBytes(32)`);
- запишет `~/.bitrix24-mcp-bridge/config.json`;
- положит стартовый `~/.bitrix24-mcp-bridge/actions.json` (каталог методов на все инструменты);
- материализует расширение в `~/.bitrix24-mcp-bridge/extension/` (бандлы + `config.json` + `manifest.json`);
- напечатает ручные шаги (см. ниже).

**Мультипортальность:** просто запусти `setup` ещё раз и выбери `[a]` — добавить портал.
Манифест расширения перечислит origin'ы всех сконфигурированных порталов (least-privilege на origin).

### 3. Загрузить расширение в браузер

1. `chrome://extensions` → включи **Developer mode** → **Load unpacked** →
   выбери **`~/.bitrix24-mcp-bridge/extension/`**.
2. Открой/обнови залогиненную вкладку портала (где грузится `BX` и мессенджер).

Работает только в Chromium: `ws://127.0.0.1` из HTTPS разрешён исключением для loopback.

### 4. Зарегистрировать MCP-сервер у агента

Секреты в конфиг агента не попадают — их читает сам мост из `~/.bitrix24-mcp-bridge/`.
Для **Claude Code**:

```bash
claude mcp add bitrix24-bridge -s user -- npx -y bitrix24-mcp-bridge
```

(Если ставил CLI глобально шагом выше — подставь `-- bitrix24-bridge` вместо `-- npx -y
bitrix24-mcp-bridge`. Для другого MCP-клиента — та же команда без аргументов.)

При старте клиент подключается к daemon по UDS, а если сокета нет — **сам поднимает daemon**
(`bitrix24-bridge --daemon`, detached). Вручную daemon запускать не нужно.

### 5. Проверка

Спроси у агента `bitrix_status`. Ожидаемо: `configured: true`, и портал с открытой
вкладкой — в списке подключённых. Если портал сконфигурирован, но не подключён — открой
залогиненную вкладку. Дополнительно — `docs/e2e-checklist.md`.

## Повседневная работа

После разовой настройки живая работа требует немного:

1. **Быть залогиненным** в Bitrix24 с открытой обычной вкладкой портала (расширение
   подключается само).
2. **Запустить агента** — он поднимет daemon сам. Вручную ничего запускать не надо.
3. Пользоваться инструментами: `bitrix_tasks_list`, `bitrix_task_get`,
   `bitrix_projects_list`, `bitrix_project_get`, `bitrix_chats_recent`,
   `bitrix_chat_messages`, а также `bitrix_call` для вызовов из каталога, ещё не
   обёрнутых типизированным инструментом. У каждого — необязательный `portal` (alias).

Что **не** нужно каждый раз: пересобирать расширение, пересоздавать `actions.json`, менять токен.

## Правки конфигурации (`bitrix24-bridge setup` → меню)

Повторный `setup` открывает меню правки:

| Клавиша | Действие | Что перезагрузить после |
|---|---|---|
| `a` | добавить портал | **перезагрузить расширение** (`chrome://extensions`) |
| `r` | удалить портал | перезагрузить расширение |
| `e` | изменить origin портала | перезагрузить расширение |
| `d` | сменить портал по умолчанию | ничего (меняется только config) |
| `p` | сменить порт | **переоткрыть вкладку** портала |
| `t` | ротировать токен | переоткрыть вкладку портала |
| `u` | обновить расширение из установленного пакета | перезагрузить расширение |
| `q` | выход | — |

**daemon перезапускается автоматически.** Работающий daemon читает `config.json` только при
старте, поэтому `setup` сам останавливает его после любой правки, затрагивающей конфиг (порт,
токен, набор порталов) — следующий вызов агента поднимает новый daemon с актуальным конфигом; вручную
ничего делать не нужно. Если daemon по какой-то причине не остановился (например, `setup` не смог
достучаться до UDS-сокета), заверши процесс вручную — `pkill -f -- --daemon` (или `kill` его pid по
`lsof -ti tcp:<порт из config.json>`) — это лишь фолбэк на нештатный случай.

После рестарта daemon дополнительно: **изменение набора порталов** правит `manifest.json` → нужна
перезагрузка расширения в `chrome://extensions`; **изменение порта/токена** правит `config.json`
расширения → переоткрой вкладку портала.

## Расширение охвата (добавить календарь / диск / CRM / новый вызов)

Конечному пользователю код менять не нужно — это работа с данными в
`~/.bitrix24-mcp-bridge/actions.json`:

1. Сними HAR по нужному разделу — см. **[docs/reconnaissance.md](reconnaissance.md)**.
2. Добавь запись с понятным ключом (`endpoint`/`action`/`method`/`params`/`bodyType`).
3. Перезапусти MCP-клиент (или заверши daemon — `pkill -f -- --daemon`), чтобы он перечитал каталог.
4. Вызови через `bitrix_call { name: "<ключ>", params: {…} }`.

Типизированные обёртки поднимаются для тех ключей каталога, что они ждут (задачи, проекты, чаты,
поиск — полный список в README и `bitrix_help`); недостающие ключи просто не регистрируют свой
инструмент. Каталог может включать мутирующие вызовы — allowlist (факт наличия записи) — единственная
граница.

**Мейнтейнеру (правка в репозитории).** Каталог и код сервера живут в репо, а daemon читает
runtime-папку (`~/.bitrix24-mcp-bridge/actions.json` + собранный `dist/cli.js`) — правки репо сами
не доезжают. После изменения репо-`actions.json` или `src/tools/register.ts` выполни
**`bun run sync:runtime`**: пересборка бандла + копия каталога в runtime-папку + гашение daemon.

## Диагностика

| Симптом | Причина | Что делать |
|---|---|---|
| Агент видит только `bitrix_help`/`bitrix_status`, `status` = `configured:false` | нет `config.json` | выполни `bitrix24-bridge setup` |
| `bitrix_status`: портал не подключён | нет открытой залогиненной вкладки | открой вкладку портала |
| daemon не стартует, ошибка чтения `actions.json` | битый JSON | поправь каталог |
| соединение расширения закрывается сразу | неверный токен или Origin вне набора порталов | сверь `config.json` расширения; после смены токена — переоткрой вкладку |
| ответ содержит `invalid_csrf` / `invalid_authentication` | сессия протухла | перелогинься в браузере, переоткрой вкладку |
| `session context not ready` | вкладка без `BX` (встроенное приложение) | открой обычную вкладку портала |
| после `setup` правки вызовы не проходят / расширение не коннектится | `setup` не смог остановить старый daemon (нештатный случай) | заверши daemon вручную (`pkill -f -- --daemon`) — новый поднимется с актуальным config |
| после add/remove/edit портала вызовы к нему не идут | не перезагружено расширение | `chrome://extensions` → обнови расширение |
| инструмент возвращает пусто/ошибку Bitrix | неверные `endpoint`/`action` в `actions.json` | сверь с реальным запросом в DevTools, поправь **данные** |
| агент не видит новые методы/инструменты, хотя в репо они есть | daemon отстал: старый бандл или старый `~/.bitrix24-mcp-bridge/actions.json` | мейнтейнеру — `bun run sync:runtime` (пересборка + копия каталога + перезапуск daemon) |

Ручной запуск для отладки (dev, из репозитория): `bun run src/index.ts --daemon` — увидишь
логи daemon на stderr; `bun run src/index.ts` — MCP-клиент.

## Безопасность (кратко)

- daemon слушает только loopback (`127.0.0.1`) и требует общий токен → держи токен длинным
  и случайным (`setup` генерирует `crypto.randomBytes(32)`).
- daemon отклоняет WS-подключения, чей браузерный Origin не входит в набор сконфигурированных порталов.
- Каталог = allowlist: вызвать можно только описанное — включая мутирующие вызовы, если они есть в каталоге.
- Токен и WebSocket живут **только в ISOLATED-мире** расширения; страница портала не может
  прочитать `config.json` расширения из-за `use_dynamic_url`. MAIN-шим отдаёт лишь `sessid`,
  который у страницы и так есть.
- UDS `bridge.sock` создаётся с правами `0600` — граница здесь обеспечивается ФС.
- Подробнее — [`../README.md`](../README.md) и [`../extension/README.md`](../extension/README.md).
