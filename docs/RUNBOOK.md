# Runbook — Bitrix24 MCP Bridge

Как запустить мост первый раз и как пользоваться им дальше.

Мост даёт локальному ИИ-агенту **read-only** доступ к твоим задачам, проектам и чатам Bitrix24 —
в объёме прав твоего пользователя, через **твою уже открытую браузерную сессию**. Без прав
администратора и без официального REST-вебхука.

---

## Как это устроено (чтобы был понятен порядок запуска)

```
ИИ-агент ──stdio──► MCP-сервер ──WS (127.0.0.1:39917, токен + Origin)──► Расширение (вкладка портала)
                                                                            │  fetch + свежий sessid + cookie
                                                                            ▼
                                                                Bitrix24 (задачи / группы / чаты)
```

Для живого запроса нужны **одновременно** три вещи:
1. запущенный MCP-сервер (его обычно поднимает сам агент),
2. открытая **залогиненная** вкладка портала,
3. установленное расширение на этой вкладке.

Ключевые принципы:
- **`actions.json` — это данные и allowlist одновременно.** Что не описано в нём — вызвать нельзя.
- **Read-only enforced кодом:** каталог с мутирующим вызовом (`…add`/`…update`/`…delete`/…) не загрузится.
- **Мост слушает только `127.0.0.1`**, требует общий токен и совпадение Origin вкладки.
- **Свежесть = момент запроса.** Подписки на новые сообщения нет.

---

## Требования

- **Bun** (проверить: `bun --version`).
- **Chromium/Chrome** (мост работает через loopback-исключение для `ws://127.0.0.1`).
- Аккаунт Bitrix24, под которым ты залогинен в браузере.

## Первый запуск (разовая настройка)

### 1. Зависимости

```bash
cd /path/to/bitrix24
bun install
bun test          # ожидаемо: 36 pass / 0 fail
```

### 2. Настроить `.env` (единая точка правды)

Вся конфигурация живёт в **`.env`** (gitignored). Его автоматически подхватывают и MCP-сервер,
и сборка расширения — руками исходники редактировать не нужно.

```bash
cp .env.example .env
# сгенерировать токен:
openssl rand -hex 32
```

Заполни `.env`:

```
BITRIX_MCP_TOKEN=<вывод openssl rand -hex 32>
BITRIX_ORIGIN=https://<твой-портал>.bitrix24.ru
BITRIX_MCP_PORT=39917       # опционально
BITRIX_CATALOG=actions.json # опционально
```

Токен — единственная защита локального порта, поэтому он должен быть длинным и случайным.
Трейлинг-слэш в `BITRIX_ORIGIN` можно не убирать — он нормализуется автоматически.

### 3. Разведка → собрать `actions.json`  ⚠️ главный ручной шаг

Сервер **не стартует без `actions.json`** (в репозитории есть только `actions.example.json`).
`action`-имена внутренних контроллеров Bitrix24 не документированы — их снимают из твоего браузера:

1. Открой вкладку своего Bitrix24, залогинься.
2. **DevTools (F12) → Network** → включи **Preserve log**, фильтр не ставь.
3. Прокликай каждый домен Триады:
   - **Задачи**: список задач + открой одну задачу.
   - **Проекты**: список рабочих групп + зайди в группу.
   - **Чаты**: открой мессенджер + один диалог + проскролль историю.
4. ПКМ по списку запросов → **Save all as HAR with content** → сохрани как `captured.har` **в корень проекта**.
5. Построй черновик каталога:
   ```bash
   bun run catalog:draft      # captured.har (в корне) → actions.draft.json (в корне)
   ```
   Скрипт исполняется из корня проекта независимо от того, откуда ты его вызвал.
   Для HAR под другим именем/путём — `bun run catalog <путь.har> > actions.draft.json`.
   Читай предупреждения в stderr: `⚠ nothing captured for "chats"…` означает, что домен не был
   прокликан — вернись в браузер и повтори по нему шаг 3.
6. Собери из черновика **`actions.json`**, присвоив понятные ключи, которые ожидают типизированные
   инструменты (можно и меньше — недостающие инструменты просто не зарегистрируются):

   | Ключ в `actions.json` | Инструмент | Назначение |
   |---|---|---|
   | `tasks.list` | `bitrix_tasks_list` | список задач |
   | `task.get` | `bitrix_task_get` | карточка задачи |
   | `projects.list` | `bitrix_projects_list` | рабочие группы |
   | `chats.recent` | `bitrix_chats_recent` | недавние диалоги |
   | `chat.messages` | `bitrix_chat_messages` | история диалога |

   Формат записи (см. `actions.example.json`):
   ```json
   {
     "tasks.list":    { "action": "tasks.task.list", "method": "POST" },
     "chats.recent":  { "endpoint": "/rest/im.recent.list", "method": "POST" }
   }
   ```
   Поля: `endpoint` (по умолчанию `/bitrix/services/main/ajax.php`), `action` (по умолчанию `null`),
   `method` (по умолчанию `POST`), `params` (по умолчанию `{}`). Транспорт воспроизводится как есть —
   часть доменов на `ajax.php?action=…`, часть на `/rest/…`.

   > ⚠️ Если положишь мутирующий вызов, сервер **намеренно откажется стартовать** (read-only-гард).

### 4. Собрать и установить расширение

Токен, порт и домen подставляются в расширение из `.env` на сборке — руками ничего не редактируешь.

```bash
bun run build:ext    # читает .env → создаёт extension/bridge-client.js и extension/manifest.json
```
1. `chrome://extensions` → включи **Developer mode** → **Load unpacked** → выбери папку `extension/dist/`.
2. Открой/обнови залогиненную вкладку портала.

> Сменил токен/домен/порт? Поправь `.env`, снова `bun run build:ext`, и нажми «обновить» у расширения.

### 5. Зарегистрировать MCP-сервер у агента

Сервер тоже читает `.env` (Bun подхватывает его через `--env-file`) — секреты не попадают в конфиг агента.
Для **Claude Code**:

```bash
claude mcp add bitrix24 -- \
  bun --env-file=/path/to/bitrix24/.env \
      run /path/to/bitrix24/src/index.ts
```

(Для другого MCP-клиента — тот же смысл: команда `bun --env-file=<abs>/.env run <abs>/src/index.ts`.
Либо, если предпочитаешь, передай значения через механизм env самого клиента.)

Переменные (все в `.env`):

| Переменная | Обязательна | По умолчанию | Назначение |
|---|---|---|---|
| `BITRIX_MCP_TOKEN` | да | — | общий токен (тот же, что вшит в расширение при сборке) |
| `BITRIX_ORIGIN` | да | — | `https://<портал>.bitrix24.ru`; проверка Origin вкладки (слэш в конце нормализуется) |
| `BITRIX_MCP_PORT` | нет | `39917` | порт WS-моста |
| `BITRIX_CATALOG` | нет | `actions.json` | путь к каталогу |

### 6. Проверка

Пройди `docs/e2e-checklist.md`. Быстрый ручной старт сервера (без агента) — `.env` подхватится сам:

```bash
bun run src/index.ts
# stderr: [bridge] listening on 127.0.0.1:39917
# после открытия вкладки портала: [bridge] extension authenticated
```

## Последующие запуски (повседневно)

После разовой настройки живая работа требует немного:

1. **Быть залогиненным** в Bitrix24 в браузере, с открытой обычной вкладкой портала
   (расширение подключится само, автопереподключение каждые 3 с).
2. **Запустить агента.** Если MCP-сервер зарегистрирован через `claude mcp add`, агент
   **поднимает сервер сам** при старте — вручную ничего запускать не нужно.
3. Пользоваться инструментами: `bitrix_tasks_list`, `bitrix_projects_list`, `bitrix_chats_recent`,
   `bitrix_chat_messages`, `bitrix_task_get`, а также универсальный `bitrix_call` для вызовов из
   каталога, ещё не обёрнутых типизированным инструментом.

Что **не** нужно каждый раз: пересобирать расширение, пересоздавать `actions.json`, менять токен.

Когда нужно вмешаться:
- **Перелогинился/сменил пароль** → просто открой вкладку заново, сессия подхватится.
- **Обновил `actions.json`** (добавил домен) → перезапусти MCP-сервер (в Claude Code — перезапуск клиента/переустановка MCP-записи). Расширение трогать не надо.
- **Поменял токен / домен / порт** → правишь только `.env`, затем `bun run build:ext` и «обновить» у расширения. Сервер перечитает `.env` при следующем старте — исходники не трогаешь.

## Расширение охвата (добавить календарь / диск / CRM / новый вызов)

Код менять не нужно — это работа с данными:

1. Сними HAR по нужному разделу (как в шаге 3).
2. Добавь запись в `actions.json` с понятным ключом.
3. Вызови через `bitrix_call { name: "<ключ>", params: {…} }`.
4. Перезапусти MCP-сервер, чтобы он перечитал каталог.

(Только read-only вызовы — мутирующие каталог отвергнет.)

## Диагностика

| Симптом | Причина | Что делать |
|---|---|---|
| Сервер не стартует, `looks mutating` | в `actions.json` мутирующий вызов | убери его — мост read-only |
| Сервер не стартует, `is required` | не задан `BITRIX_MCP_TOKEN`/`BITRIX_ORIGIN` | задай env |
| Сервер не стартует, ошибка чтения `actions.json` | файла нет / битый JSON | собери каталог (шаг 3) |
| Порт занят (`EADDRINUSE`) | сервер уже запущен другим процессом | останови старый или смени `BITRIX_MCP_PORT` |
| `extension not connected` | нет открытой вкладки портала / расширение не собрано | открой вкладку; `bun run build:ext`; сверь токен |
| `session context not ready` | вкладка без `BX` (встроенное приложение) | открой обычную вкладку портала |
| ответ содержит `invalid_csrf` / `invalid_authentication` | сессия протухла | перелогинься в браузере |
| соединение расширения закрывается сразу | неверный токен или Origin ≠ `BITRIX_ORIGIN` | сверь токен в обоих местах и домен |
| инструмент возвращает пусто/ошибку Bitrix | неверные `endpoint`/`action` в `actions.json` | сверь с реальным запросом в DevTools, поправь **данные** (код не трогай) |

## Безопасность (кратко)

- Мост слушает только loopback (`127.0.0.1`) и требует общий токен → используй длинный случайный токen.
- Сервер отклоняет WS-подключения, чей браузерный Origin ≠ `BITRIX_ORIGIN`.
- Каталог = allowlist: вызвать можно только то, что в нём есть; мутирующие вызовы отвергаются на загрузке.
- Расширение работает в `world: MAIN`, поэтому скрипт, уже исполняющийся на твоей залогиненной
  странице портала, теоретически может прочитать токен — но такой скрипт и так уже владеет твоей
  сессией, так что эскалация ограничена.
- См. также `extension/README.md`.
```
